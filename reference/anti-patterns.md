# review 反模式：5 大常见问题 + Why/How

> 本文档用于指导 AI 在生成复盘报告时识别并规避常见问题的本质原因，
> 在边缘场景做出正确判断。每条都包含：**是什么** + **Why**（为什么不行）+ **How**（怎么做才对）。

---

## 1. 凭记忆填写

**现象**：不查 Git、不查 Traces，直接凭印象写"今天完成了 XX、YY、ZZ"。

**Why**：
- 记忆有滞后和选择性——人会强化成就感，淡化失败；会把做了一半的事说成"完成"
- 复盘最大的价值在于找出**自己没意识到的偏差**。凭记忆永远找不出来
- 月报、季报、年报若基于印象写，方向判断会系统性失真

**How**：
- 日报：必跑 `collect-git-data.sh` + `analyze-traces.sh`
- 周报及以上：必读前一周期的所有子报告（日报→周报→月报→季报→年报）
- 数字来自脚本，叙事来自数据，不从记忆编造

**边缘场景**：
- 会议、讨论、线下沟通这些不进 git —— 靠`💼工作项目/总结报告/会议纪要/` 作为第二数据源
- 如果确实没有会议纪要，在报告里明确标注"此部分来自回忆"，提醒自己下次要记

---

## 2. 泛泛而谈

**现象**：`完成了产品优化工作`、`推进了项目进度`、`解决了几个问题`。

**Why**：
- 泛泛表述无法追溯——3 个月后再看这份报告，你不知道"几个问题"具体是什么
- 没有具体案例就没有模式识别。月报/季报做模式识别的前提是周报足够具体
- 没有数字就没有量化——`解决了几个` vs `解决了 4 个 P0 bug，累计节省 6 小时`，一个是口号，一个是数据

**How**：
- 每条工作必须有：**数字**（几个/几小时/几行）+ **具体对象**（什么文件/什么 bug/什么页面）+ **结果**（解决/未解决/部分解决）
- 写"XX 产品优化" → 改写为"XX 产品的搜索页性能优化，首屏从 2.1s → 0.8s（验证：Lighthouse 截图）"

**边缘场景**：
- 不敢写数字是因为没量化 —— 先量化再写，没条件量化就标注"待量化"
- 涉及保密信息，用代号替换但不去掉数字 —— 比如"客户 A 的方案从 3 周 → 1 周"

---

## 3. 只说问题不给方案

**现象**：`本周反复修改搜索接口 5 次`、`Token 消耗偏高` —— 下一句就完了，没说下一步怎么办。

**Why**：
- 复盘的定义是"下次不再犯"。不给方案等于把问题原样保留到下周期
- 周报里只说问题，月报做"模式识别"时会发现同一问题已经说了 4 周，每周都在抱怨但没解决——这是系统性失败
- 改进动作是复盘最贵的产出。不写下来，团队/个人没有行动点

**How**：
- 每个 ⚠️ / 🔴 问题后面必须跟 3 个字段：
  - **根因**（5 问法）：为什么会发生？—— 深挖到"因为没有 X 规则"
  - **动作**：具体做什么（改哪个文件 / 加哪条规则 / 谁负责）
  - **何时**：这周内 / 下月初 / 季度内
- 如果当下想不清方案，至少写"下周用 1 小时专门想清楚，deadline YYYY-MM-DD"

---

## 4. 忽略反复修改

**现象**：看到 git log 里同一文件连续 5 次 commit，忽略过去，当成"正常迭代"。

**Why**：
- 反复修改是效率损失的最大黑洞——同一 bug 改 3 次 = 原本 1 次工作量 × 3 + 心理成本
- "反复修改"和"正常迭代"的区别在 commit message：含 fix/修复/回滚/revert/改回 = 返工；含 feat/优化 = 迭代
- 忽略反复修改 → 一个月后同样的问题还在同样的文件里重现，形成"永远修不完"的错觉
- 这是 AI 协作场景特别典型的问题：AI 改一半没改完，下一轮又改错方向

**How**：
- 日报/周报强制跑 `detect-repeated-changes.sh`
- 阈值：同一文件 ≥ 3 次改动 + 包含返工关键词 → 标记 🔴，必须做根因分析
- 根因分析模板：
  - 第一次改：为什么没想清楚？
  - 第二次改：第一次的方案哪里错了？
  - 第三次改：是不是一开始就应该换方案？
  - 本质规则：下次遇到类似场景，应该怎么做第一步？

**边缘场景**：
- 原型探索期允许快速迭代，不算返工 —— 脚本按 commit 关键词区分，不需要人工判断
- 真正的 refactor 不是返工 —— commit message 用 "refactor:" 前缀区分

## 5. 混淆用户 vs Claude 投入

**现象**：日报写"今天完成 5 小时 harness 构建" —— 实际是 Claude 执行了 5 小时工具调用，用户只花了 30 分钟决策。

**Why**：
- AI 协作时代，**工具调用时长 ≠ 人的工作时长**
- 混淆归属会导致两类错误：
  - **高估个人产出**："我今天写了 2000 行代码" —— 其实 1900 行是 Claude 生成
  - **低估 AI 价值**："Claude 只辅助一点点" —— 实际 Claude 独立完成了 80%
- 长期混淆 → 月报季报的时间分配分析完全失真，错误决策投入方向

**How**（深版日报强制）：
- 所有时长/产出必须三选一标注：`[用户]` / `[Claude]` / `[协作]`
- 区分表：

  | 归属 | 定义 | 示例 |
  |---|---|---|
  | **用户** | 亲手写、亲手改、亲自决策、亲自审阅 | 手敲 commit、IDE 改文件、决定是否合并 |
  | **Claude** | AI 通过 Read/Edit/Write/Bash 工具产生 | 生成的 hook 脚本、自动重构的代码 |
  | **协作** | 完整对话闭环：提需求 → 执行 → 审阅 | 一次对话的端到端时长 |

- 时长分开报告：`用户 30 分钟 / Claude 4 小时 / 协作时长 1.5 小时`
- commit 列表区分：本人手写 vs Claude 执行

**边缘场景**：
- 用户让 Claude 写代码，然后自己大改 —— 按主体贡献归类：改动 > 50% 算用户，否则算协作
- Claude 跑长期后台任务（如 loop） —— 不计入用户时长，只计入 Claude 执行时长

---

## 判断框架：遇到边缘场景时怎么办？

1. **分不清是否反复修改？** —— 跑 `detect-repeated-changes.sh`，脚本说了算，不人工判断
2. **写不出具体数字？** —— 先跑 `collect-git-data.sh`，数字都在 git log 里
3. **分不清用户 vs Claude？** —— 看这行代码是**谁按下的 Enter**：人 = 用户，Claude 工具调用 = Claude
4. **问题暂时想不清方案？** —— 至少写"Parking Lot：YYYY-MM-DD 前 1 小时专项思考"
5. **没时间做完整复盘？** —— 用日报简版，别跳过。跳过一天，周报就断一天
