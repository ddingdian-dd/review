**设计理念**：从"记录发生了什么" → "分析为什么这样" → "预测会怎么样" → "指导下一步"

**基于**：AI驱动的工作报告体系设计（2026-04-24）

---

## 核心改进

### 改进 #1：统一底层结构（母版）

所有报告基于 **9 个核心模块**裁剪：

```
1. 关键结论（Executive Summary）         - 3-5 条核心结论
2. 核心数据（Metrics）                   - 量化数据（Git/Token/时间）
3. 重点工作（Execution）                 - 具体产出
4. 偏差识别（Deviation Detection）       - 反复修改、效率损失
5. 动作分级（Action Classification）     - ✅⚠️🔴 三级分类
6. 因果分析（Root Cause Analysis）       - 5 问法追问根因
7. 模式识别（Pattern Recognition）       - 重复问题识别
8. 预测性分析（Predictive Analysis）     - 趋势预测
9. 自我进化（Self-Evolution）            - Model/Harness/Context/Traces
```

---

### 改进 #2：分层递进模型

```
日报（执行层）     → 发生了什么 + 有无偏差
周报（收敛层）     → 做得怎么样 + 高效/低效动作
月报（复盘层）     → 为什么这样 + 重复问题根因
季报（策略层）     → 方向对不对 + ROI 评估
年报（决策层）     → 值不值得继续 + 能力提升
```

**关键**：每一层都基于上一层，而非孤立生成。

---

### 改进 #3：强制数据源

**禁止凭记忆填写**，必须基于以下数据源：

```
1. Git 提交记录（代码变更、提交时间、commit message）
2. Traces 执行日志（工具调用、执行步骤、成功/失败）
3. Token 消耗记录（输入/输出 token、浪费识别）
4. Session 交互记录（对话历史、问题解决过程）
5. 上一层级报告（周报基于日报，月报基于周报）
```

---

### 改进 #4：AI 独有的分析视角

**新增 5 种分析方式**：

1. **反事实分析**（Counterfactual）：如果不这样做，会怎样？
2. **边际效益分析**（Marginal Benefit）：下一个小时最应该投在哪里？
3. **学习曲线分析**（Learning Curve）：我的能力在提升吗？
4. **系统动力学分析**（System Dynamics）：正反馈循环如何放大/抑制？
5. **能力矩阵分析**（Capability Matrix）：我的短板在哪里？

## 模板体系结构

### 层级 1：日报（执行层）

**文件**：`daily-report-v2.md`

**核心目标**：记录事实，识别偏差

**模块裁剪**：
- ✅ 1. 关键结论（简化版，1-2 条）
- ✅ 2. 核心数据（Git/Token/时间）
- ✅ 3. 重点工作（具体产出）
- ✅ 4. 偏差识别（反复修改、效率损失）
- ❌ 5. 动作分级（周报才需要）
- ❌ 6. 因果分析（周报才需要）
- ❌ 7. 模式识别（月报才需要）
- ❌ 8. 预测性分析（季报才需要）
- ❌ 9. 自我进化（月报才需要）

**强制数据源**：
```bash
# Git 数据
git log --since="今日 00:00:00" --until="今日 23:59:59" --numstat

# Token 数据（从 Traces 提取）
~/.claude/traces/*/trace.json

# 工作时长（从 Git 提交时间戳计算）
git log --since="今日 00:00:00" --format="%at"
```

### 层级 2：周报（收敛层）

**文件**：`weekly-report-v2.md`

**核心目标**：汇总成果，识别高效/低效动作

**模块裁剪**：
- ✅ 1. 关键结论（3-5 条）
- ✅ 2. 核心数据（周度汇总）
- ✅ 3. 重点工作（基于日报汇总）
- ✅ 4. 偏差识别（汇总本周所有偏差）
- ✅ 5. 动作分级（✅⚠️🔴 三级分类）
- ✅ 6. 因果分析（5 问法追问根因）
- ❌ 7. 模式识别（月报才需要）
- ❌ 8. 预测性分析（季报才需要）
- ⚠️ 9. 自我进化（简化版，只看 Context 层）

**强制流程**（最重要）：
```
Step 1: 读取本周所有日报（绝对优先）
  - 查找 `总结报告/日报/` 目录下本周的所有日报文件
  - 逐个读取日报内容
  - 提取关键信息：完成任务、偏差识别、Token消耗、教训

Step 2: 采集 Git 数据（补充日报未记录的信息）
  - git log --all --since="本周一" --until="本周日"
  - 统计代码变更量、提交次数、提交者分布

Step 3: 对比分析
  - 日报记录 vs Git 实际提交（检查是否有遗漏）
  - 汇总本周关键动作、踩坑、经验沉淀

Step 4: 生成周报
  - 基于日报内容生成周报
  - 保证周报和日报一致性
```

**新增模块**：
- **动作分级**：✅ 高效动作（量化收益）/ ⚠️ 可优化动作（量化损失）/ 🔴 禁止动作
- **因果分析**：5 问法追问根因
- **反事实分析**：如果不建立 XX 规则，会多花费多少成本？

- ✅ 3. 重点工作（基于周报汇总）
- ✅ 4. 偏差识别（月度趋势）
- ✅ 5. 动作分级（月度汇总）
- ✅ 6. 因果分析（深度根因分析）
- ✅ 7. 模式识别（重复问题识别）
- ⚠️ 8. 预测性分析（简化版，趋势分析）
- ✅ 9. 自我进化（Context + Harness 层）

**强制流程**：
```
Step 1: 读取本月所有周报（绝对优先）
  - 查找 `总结报告/周报/` 目录下本月的所有周报文件
  - 提取每周的关键结论、问题、经验

Step 2: 选择性读取日报（如周报不完整）
  - 如果周报信息不足，补充读取日报

Step 3: 采集 Git 数据（补充）

Step 4: 生成月报
  - 强调本月的因果关系
  - 识别重复出现的问题（在多个周报中出现）
```

**新增模块**：
- **模式识别**：重复问题清单（问题 + 出现次数 + 累计损失）
- **根因分析树**：问题 → 原因 → 原因的原因 → 根因
- **改进措施验证**：A/B 对比（改进前 vs 改进后）
- **学习曲线分析**：3 个维度的能力提升趋势

### 模块 6：因果分析（Root Cause Analysis）

**目标**：5 问法追问根因

**格式**：
```markdown
## 🔍 因果分析

### 问题：UI 反复修改 7 次

**5 问法分析**：

**Q1：为什么 UI 反复修改 7 次？**
→ A1：因为没有最终效果图，凭感觉调整

**Q2：为什么没有最终效果图？**
→ A2：因为觉得"画图太慢，直接改代码更快"

**Q3：为什么觉得画图太慢？**
→ A3：因为没有意识到反复修改的成本（3-4h）远高于画图成本（0.5h）

**Q4：为什么没有意识到反复修改的成本？**
→ A4：因为没有量化效率损失

**Q5：为什么没有量化效率损失？**
→ A5：因为没有建立 Traces 系统，无法自动统计

**根因**：缺少 Traces 系统 → 无法量化效率损失 → 缺少成本意识 → 低效决策

**系统性解决方案**：
1. 短期：建立"重要 UI 变更前画 Figma"的规则
2. 中期：建立 Traces 系统，自动量化效率损失
3. 长期：建立成本意识文化
```

### 模块 7：模式识别（Pattern Recognition）

**目标**：识别重复问题（在多个报告中出现 ≥2 次）

**格式**：
```markdown
## 🔄 模式识别

### 重复问题清单（本月出现 ≥2 次）

| 问题 | 出现次数 | 累计损失 | 是否需要系统性解决 |
|------|---------|---------|------------------|
| UI 反复修改 | 3 次 | ~10h | ✅ 是（建立 Figma 确认流程） |
| Token 浪费 | 4 次 | ~$70 | ✅ 是（建立 Token 优化规则） |
| 配置管理混乱 | 2 次 | ~6h | ✅ 是（建立配置文档管理规范） |

---

### 深度分析：UI 反复修改

**出现记录**：
1. 第 1 周：result-ai-report.html 修改 5 次（损失 3h）
2. 第 2 周：homepage.html 修改 3 次（损失 2h）
3. 第 4 周：result-ai-report.html 修改 7 次（损失 3-4h）

**趋势分析**：
- 频率：持续出现（3/4 周）
- 损失：累计 ~10h
- 严重性：⚠️ 高（每次损失 2-4h）

**根因**（基于模块 6 的因果分析）：
- 直接原因：缺少最终效果图
- 深层原因：缺少成本意识
- 根本原因：缺少 Traces 系统

**系统性解决方案**：
1. ✅ 已建立规则："重要 UI 变更前画 Figma"
2. ⚠️ 待实施：建立 Traces 系统
3. ⚠️ 待观察：下月是否还会出现

**验证计划**：
- 下月继续监控
- 如果仍出现 ≥2 次 → 规则执行不力，需要强化
- 如果降到 0-1 次 → 规则有效，继续执行
```

### 模块 8：预测性分析（Predictive Analysis）

**目标**：基于历史数据预测未来趋势

**格式**：
```markdown
## 📈 预测性分析

### 预测 #1：Token 优化效果

**历史数据**：
| 日期 | Token 消耗 | 浪费率 | 趋势 |
|------|-----------|--------|------|
| 04-20 | 150,000 | 30% | 基线 |
| 04-21 | 150,000 | 42% | ⬆️ 上升 |
| 04-22 | 45,000 | 56% | ⬆️⬆️ 急升 |
| 04-23 | 30,000 | 21% | ⬇️⬇️⬇️ 优化后骤降 |

**趋势分析**：
- 优化前：浪费率持续上升（30% → 42% → 56%）
- 优化后：浪费率大幅下降（56% → 21%）
- 下降幅度：63%

**预测模型**：
```
如果继续执行 Token 优化规则：
  预计下周浪费率 = 21% ± 3%（稳定区间：18-24%）
  预计月度浪费率 = 20%
  预计年度节省 = $350

如果不执行规则：
  预计下周浪费率 = 45%（回归到优化前水平）
  预计月度浪费率 = 50%
  预计年度多花费 = $350
```

**建议**：
- ✅ 必须强制执行 Token 优化规则
- ⚠️ 建立自动化监控（实时反馈浪费率）
- ⚠️ 每周验证效果（如果浪费率 > 25% → 规则执行不力）

---

### 预测 #2：项目上线时间

**历史数据**：
| 日期 | 完成度 | 日进度 |
|------|--------|--------|
```
